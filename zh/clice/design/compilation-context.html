编译上下文 ​

背景 ​

在 C++ 中，同一个文件在不同的条件下可以产生完全不同的编译结果。对于源文件来说，不同的编译命令（-std=c++17 vs -std=c++20，不同的 -D 宏定义等）会导致不同的 AST。对于头文件来说，情况更加复杂——同一个头文件被不同的源文件包含时，由于前面的 #include 和 #define 不同，可能产生完全不同的预处理结果。

然而，现有的所有语言服务器——不仅是 C++ 的——都没有正式处理这个问题。用户在编辑器中打开一个文件时，语言服务器只会选择一个"默认"的编译上下文来提供服务。当这个默认选择不正确时，用户会看到错误的诊断、错误的补全建议、错误的跳转目标。

以 clangd 为例，社区长期存在大量相关问题：

  • 头文件打开后显示错误的诊断信息，因为 clangd 选择了错误的宿主源文件（host source）
  • 多配置项目（如同时支持 Debug/Release、不同平台）中，用户无法选择使用哪套编译命令
  • 非自包含头文件（non-self-contained headers）缺少正确的上下文，导致大量误报

clice 将编译上下文（Compilation Context）作为一等公民贯穿整个设计，从根本上解决这些问题。

什么是编译上下文 ​

编译上下文是"正确编译一个文件所需的全部条件"。根据文件类型不同，编译上下文的含义也不同：

源文件的编译上下文 ​

对于源文件（.cpp），编译上下文就是它的编译命令——从编译数据库（CDB）中读取的编译选项。一个源文件可能对应多条编译命令（例如在多配置项目中），每条命令代表一个不同的编译上下文。

这种情况相对简单：用户选择一条编译命令，即选择了一个编译上下文。

头文件的编译上下文 ​

对于头文件（.h），情况分为两种：

自包含头文件：如果头文件本身可以独立编译（有自己的 CDB 条目，且不依赖宿主源文件提供的前置 include 或宏定义），则可以当做源文件处理，直接使用其编译命令。

非自包含头文件：如果头文件依赖于包含它的源文件提供的前置上下文（前面的 #include、#define 等），则需要合成一个编译上下文。这个合成上下文由以下部分组成：

  • 宿主源文件（host source）：一个包含该头文件的源文件
  • include 链：从宿主源文件到目标头文件的 include 路径
  • 前缀代码（preamble）：宿主源文件中位于 #include "target.h" 之前的所有内容

这就是头文件上下文（Header File Context） 的概念——它本质上是编译上下文在头文件场景下的特化。

设计方案 ​

编译上下文的解析流程 ​

当一个文件需要编译时，按以下优先级确定其编译上下文：

  1. 用户主动选择：如果用户通过扩展协议（clice/switchContext）指定了编译上下文，使用用户的选择
  2. CDB 直接查找：如果文件本身在编译数据库中有条目，使用该条目的编译命令
  3. 头文件上下文合成：如果文件不在 CDB 中（典型的头文件场景），自动合成编译上下文

头文件上下文的合成 ​

头文件上下文的合成过程如下：

第一步：查找宿主源文件。 通过 DependencyGraph 的反向 include 图，BFS 向上搜索所有传递包含该头文件的源文件。选择第一个在 CDB 中有条目的源文件作为宿主。

第二步：计算 include 链。 从宿主源文件到目标头文件，通过正向 include 图找到最短路径。例如：main.cpp → utils.h → math.h，如果目标是 math.h，则 include 链为 [main.cpp, utils.h, math.h]。

第三步：合成前缀代码。 对于 include 链中的每个文件（除了最后的目标文件），读取其内容，扫描找到包含下一个文件的 #include 行，提取该行之前的所有内容，加上 #line 指令用于正确的错误定位。

例如，main.cpp 的内容是：

cpp
#include <vector>
#define DEBUG 1
#include "utils.h"
int main() { ... }

utils.h 的内容是：

cpp
#pragma once
#include <string>
#include "math.h"
void util_func();

那么为 math.h 合成的前缀代码为：

cpp
#line 1 "main.cpp"
#include <vector>
#define DEBUG 1
#line 1 "utils.h"
#pragma once
#include <string>

第四步：缓存前缀文件。 将合成的前缀代码写入磁盘缓存，使用内容的 xxh3 哈希值作为文件名，实现内容寻址的去重。缓存路径为 {cache_dir}/header_context/{hash}.h。

第五步：注入编译命令。 使用宿主源文件的 CDB 编译命令，将源文件路径替换为目标头文件路径，并通过 -include 标志注入前缀文件。最终的编译命令形如：

clang -std=c++17 -Iinclude -include cache/header_context/abc123.h -c math.h

这样，math.h 就如同在宿主源文件的正确位置被编译一样，获得了正确的预处理器状态。

编译上下文的切换 ​

clice 通过 LSP 扩展协议支持用户主动切换编译上下文：

  • 查询可用上下文（clice/queryContext）：列出一个文件所有可用的编译上下文。对于头文件，返回所有可能的宿主源文件；对于源文件，返回所有匹配的 CDB 条目。
  • 查询当前上下文（clice/currentContext）：返回用户主动选择的编译上下文（即通过 switchContext 设置的）；如果使用默认上下文则返回空。
  • 切换上下文（clice/switchContext）：用户选择一个新的编译上下文。这会清除该文件的所有缓存编译产物（PCH 引用、头文件上下文、依赖快照），并标记 AST 为脏，触发重新编译。

切换上下文是一个相对重量级的操作——它会导致重新编译。但这是正确的行为：不同的上下文意味着不同的编译结果。

与 PCH 优化的协作 ​

PCH（预编译头文件）在头文件上下文中扮演重要角色。头文件的前缀代码本质上就是一个 preamble，它可以被编译为 PCH 以加速后续编译。

PCH 文件在磁盘上以前缀内容的哈希值命名，实现文件级的内容寻址去重——相同内容的前缀只会在磁盘上存储一份 PCH 文件。但目前 PCH 缓存的查找仍以文件的 path_id 为键（每个文件独立缓存），尚未实现跨文件的缓存共享。将缓存键改为内容哈希以实现真正的跨文件 PCH 共享是一个计划中的改进。

编译上下文与索引的交互 ​

索引系统对编译上下文有天然的感知。同一个头文件在不同的编译上下文下可能产生不同的符号关系（例如，某个函数可能在一个上下文中通过条件编译被定义，而在另一个上下文中不存在）。MergedIndex 将同一文件在不同编译上下文下产生的索引数据合并存储，使得用户可以在一个统一的视图中查询跨上下文的符号信息。

核心依赖 ​

  • DependencyGraph：提供 include 关系的正向和反向查询，是确定宿主源文件和 include 链的基础
  • CompilationDatabase：提供源文件的编译命令
  • PathPool：文件路径的内部化标识
  • Session：存储每个打开文件的编译上下文状态（header_context、active_context）
  • Workspace：存储 PCH 缓存，供多个 Session 共享

设计决策与权衡 ​

为什么用 -include 注入而不是修改文件内容？ 使用 -include 标志可以让 Clang 自然地处理前缀代码，不需要修改目标文件的缓冲区。这保持了文件内容的一致性，同时 #line 指令确保了诊断信息指向正确的原始文件和行号。

为什么不使用"编译宿主源文件然后在 include 处停止"的方案？ 这个方案（即直接编译宿主源文件，在解析到目标头文件时设置 hook 停止）虽然概念上更简单，但存在一个根本性问题：它不支持 PCH 优化。由于是以宿主源文件为编译主体，很难细粒度地做 preamble 分割和 PCH 缓存。而对于频繁编辑的文件，PCH 缓存是性能的关键。

为什么选择最短 include 链？ 多条可能的 include 链中，最短的链产生最少的前缀代码，意味着更小的 PCH、更快的编译。同时，更短的链通常意味着更直接的包含关系，选择它作为默认上下文更符合直觉。

已知限制 ​

  • 递归包含：如果 include 图中存在循环（通过 include guard 或 #pragma once 打断），前缀合成需要正确处理去重。目前的依赖扫描已经处理了循环，但前缀合成路径尚未完全验证这一场景。
  • 宿主选择的确定性：当多个宿主源文件可用时，当前选择第一个找到的。更好的策略可能是优先选择与目标头文件"关系最近"的宿主（例如同目录下的源文件），但这需要启发式规则。
